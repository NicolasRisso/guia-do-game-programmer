O que é a programação de jogos?

  • A programação de jogos é a área responsável por dar vida às ideias e conceitos criados pelos game designers.
  • É a disciplina que traduz as regras, mecânicas e sistemas do jogo em código, permitindo que os jogadores interajam com o mundo virtual.
  • Um bom programador de jogos não apenas domina a lógica e a tecnologia, mas também compreende os princípios de design para criar uma experiência de jogo fluida, otimizada e divertida.
  • Além de programar o próprio jogo em si, Game Engineers (Sinônimo de Game Programmer) tambem são responsáveis por desenvolver ferramentas para uso interno da equipe. Como uma funcionalidade de gerar NPCs aleatórios no mapa que seguem uma rotina, por exemplo.

Lógica de Programação

  • Lógica de programação para jogos é a base que define como o jogo “pensa”: regras, condições, consequências e fluxo das ações.
  • Ela organiza decisões (if/else), repetições (loops), estados (ex.: parado, correndo, atacando) e eventos (ex.: levou dano, colidiu) para transformar mecânicas em comportamento real.
  • Com boa lógica, o jogo fica previsível para o computador e divertido/consistente para o jogador: menos bugs, sistemas claros e fácil evolução de features.
  • Antes do código final, descreva a solução em pseudocódigo e fluxos simples (passo a passo). Isso evita retrabalho e ajuda no planejamento.

Áreas da lógica para focar

  • Fluxo de controle: if/else, switch, while/for, early return.
  • Estruturas de dados: arrays/listas, filas/pilhas, dicionários (map), e quando usar cada uma.
  • Máquinas de estados (FSM): trocar entre estados do personagem/inimigo sem “spaghetti code”.
  • Eventos e mensagens: disparar e reagir a ações (input, colisões, triggers) sem dependências fortes.
  • Tempo e atualização: uso correto de delta time, timers e ordens de execução (tick/update).
  • Matemática básica de jogos: vetores, ângulos, distâncias, interpolações (lerp), aleatoriedade controlada.
  • Arquitetura simples: separar responsabilidades (ex.: lógica, visual, dados) para facilitar manutenção.

O que é importante saber

  • Decompor problemas: dividir a mecânica em passos pequenos (detectar, decidir, agir, verificar).
  • Estados + transições claras: defina quando entra/sai de um estado e o que cada estado pode fazer.
  • Determinismo onde importa: resultados consistentes para salvar/reproduzir, especialmente em multiplayer.
  • Depuração: logs, asserts, visualização de colisões/paths e testes unitários simples.
  • Complexidade: evite loops aninhados desnecessários; prefira buscas filtradas e cache de resultados.
  • Dados dirigindo comportamento: parametrizar valores (velocidades, danos, cooldowns) fora do código.
  • Iteração segura: começar simples, validar, otimizar depois (profiling antes de micro-otimizações).

Conceitos Básicos e Paradigmas

  • Conceitos básicos de programação são os fundamentos que permitem ao computador executar instruções em sequência ou de forma condicional.
  • Com eles, criamos algoritmos — passo a passo de ações — que podem ser aplicados em qualquer área, inclusive no desenvolvimento de jogos.
  • Paradigmas de programação são estilos ou modelos que orientam a forma como escrevemos e organizamos o código.
  • Dominar os conceitos e compreender os paradigmas é essencial para escolher a melhor abordagem em cada projeto.

Áreas da lógica para focar

  • Variáveis e tipos de dados: inteiros, reais, booleanos, strings.
  • Estruturas de controle: if/else, switch, loops (for/while).
  • Funções e modularização: dividir o problema em blocos reutilizáveis.
  • Estruturas de dados: listas, arrays, dicionários e pilhas/filas.
  • Abstração e decomposição: quebrar problemas complexos em partes menores.

Principais paradigmas

  • Imperativo: foco em instruções passo a passo (ex.: C, Python em modo básico).
  • Orientado a Objetos (OO): organiza o código em classes e objetos com atributos e métodos (ex.: Java, C#, C++).
  • Funcional: prioriza funções puras e imutabilidade (ex.: Haskell, mas também usado em Python, JavaScript).
  • Lógico: baseado em regras e inferências (ex.: Prolog).
  • Estruturado: programação organizada em blocos, evitando o uso excessivo de goto.

O que é importante saber

  • Pensamento lógico: clareza no raciocínio para transformar ideias em código.
  • Organização: separar responsabilidades no código, facilitando manutenção.
  • Boas práticas: nomear variáveis de forma clara, comentar trechos importantes e evitar repetições desnecessárias.
  • Paradigma certo para o problema: saber escolher se é melhor usar OO, funcional ou imperativo em determinada situação.
  • Iteração e testes: desenvolver em pequenos passos, testando sempre para garantir resultados corretos.

Matemática e Física

  • Matemática e Física para jogos fornecem as bases para simular o mundo virtual de forma realista ou estilizada.
  • Embora muitas vezes sejam negligenciadas por quem estuda para se tornar um programador, o conhecimento de formulas físicas e de equações faz parte de um sólida fundação para ser um programador de jogos de qualidade.
  • São aplicadas em movimento, colisões, iluminação, inteligência artificial e até em sistemas de partículas.
  • Dominar esses conceitos permite criar experiências mais imersivas, eficientes, previsíveis e otimizadas.

O que estudar em Matemática

  • Álgebra: equações, funções, manipulação de variáveis.
  • Geometria analítica: pontos, vetores, retas, ângulos e planos.
  • Trigonometria: seno, cosseno, tangente — fundamentais para rotação, ondas e direção.
  • Matrizes e transformações: translação, rotação, escala; base da renderização 2D/3D.
  • Interpolação: linear (lerp), curvas de Bezier e easing functions para animações.
  • Probabilidade e estatística: geração de números aleatórios, sistemas de loot, IA estocástica.

O que estudar em Física

  • Cinemática: velocidade, aceleração, movimento uniformemente variado.
  • Dinamica: leis de Newton, forças, torque, impulso e momentum.
  • Gravidade: simulação de quedas, saltos e trajetórias parabólicas.
  • Colisões: detecção e resposta (elástica, inelástica, triggers).
  • Atrito e resistência: simular superfícies, vento e líquidos.
  • Ondas e vibrações: sons, efeitos de água, simulações avançadas.

Importância nos jogos

  • Realismo: recriar o comportamento natural do mundo físico.
  • Gameplay: criar regras consistentes que o jogador entende intuitivamente.
  • Estilização: adaptar as leis da física para jogos arcade ou cartunescos.
  • Otimização: usar simplificações matemáticas para reduzir custo computacional.

Ferramentas mais usadas

  • Ferramentas mais usadas são essenciais para transformar ideias em jogos, oferecendo motores, editores e ambientes de programação.
  • Conhecer as principais opções ajuda a escolher a melhor de acordo com o projeto e a equipe.

Principais Ferramentas

  • Unreal Engine: motor gráfico robusto, ideal para jogos AA ou AAA, com realismo gráfico, ou qualquer projeto 3D em geral. É a engine padrão do mercado, considerada de alta qualidade.
  • Unity: motor versátil, muito usado em mobile e indie games.
  • Godot: open source, leve e flexível, ótimo para projetos independentes.
  • Visual Studio: ambiente de desenvolvimento completo para C++, C#, entre outras linguagens.
  • Rider: IDE da JetBrains focada em produtividade, ótima para Unreal e C++ (Fica a dica pra quem for usar a Unreal).

O que são Engines?

  • Engines de jogos são plataformas que fornecem as ferramentas e recursos necessários para desenvolver jogos digitais.
  • Elas reúnem gráficos, física, som, inteligência artificial (No sentido de algoritmos inteligentes, não o conceito de IA generativa que está em alta hoje) e sistemas de entrada em um único ambiente.
  • Com uma engine, o desenvolvedor não precisa reinventar a roda — já existem sistemas prontos para renderização, colisões e gerenciamento de cenas.
  • Isso acelera a produção, reduz custos e permite focar mais na criatividade e no design do jogo.

Principais funções de uma Engine

  • Renderização: gerar imagens 2D e 3D em tempo real.
  • Física: simular colisões, gravidade, forças e movimento.
  • Áudio: reproduzir sons, música e efeitos.
  • Scripts: programar comportamentos e lógicas do jogo.
  • Editor visual: organizar cenas, personagens e objetos sem mexer diretamente no código.
  • Ferramentas extras: animação, inteligência artificial, rede/multiplayer e otimização.

Por que usar uma Engine?

  • Produtividade: mais rápido criar jogos com sistemas prontos.
  • Acessibilidade: reduz a barreira de entrada para iniciantes.
  • Escalabilidade: funciona tanto para jogos simples quanto para projetos AAA.
  • Comunidade: grande suporte e documentação.

Arquitetura de Software

  • Arquitetura de software é a forma como organizamos e estruturamos o código e os sistemas de um jogo para que sejam claros, escaláveis e fáceis de manter.
  • Ela define como os módulos se comunicam, separa responsabilidades e ajuda a evitar código bagunçado (spaghetti code).

Arquiteturas mais usadas em Game Programming

  • Programação Estruturada: fluxo simples, baseado em funções e procedimentos. Boa para jogos pequenos.
  • Orientada a Objetos (OOP): organiza o código em classes e objetos, muito usada em Unity (C#) e Unreal (C++).
  • Component-Based: divide comportamentos em componentes que podem ser adicionados e removidos de objetos (padrão usado em Unity e Unreal).
  • Entity-Component-System (ECS): separa dados (entidades), comportamentos (sistemas) e propriedades (componentes). Focado em performance e usado em jogos de grande escala.
  • Event-Driven: módulos se comunicam por eventos e mensagens, reduzindo dependências diretas entre classes.

Inteligência Artificial

  • Inteligência Artificial (IA) em jogos não é a mesma coisa que IA generativa (como ChatGPT ou modelos de imagem). Aqui, trata-se de técnicas focadas em controlar inimigos, NPCs e sistemas do jogo.
  • O objetivo não é criar “consciência”, mas sim simular decisões e comportamentos que pareçam inteligentes para o jogador.
  • A IA de jogos precisa ser previsível, ajustável e eficiente, para manter o equilíbrio e a diversão.

Principais técnicas

  • Finite State Machine (FSM): estados como patrulha, perseguir, atacar, fugir, com transições claras.
  • Behaviour Trees: árvores de decisão que organizam e priorizam ações de forma modular.
  • Pathfinding: algoritmos como A* e Dijkstra para encontrar rotas em grids ou navmeshes.
  • Sistemas de Percepção: visão, audição e proximidade para detectar o jogador e reagir.
  • Decision Making: regras condicionais e Utility AI (escolher a melhor ação com base em pontuação).
  • PCG (Procedural Content Generation): uso de algoritmos para gerar mapas, fases, inimigos ou itens dinamicamente, aumentando a rejogabilidade.
  • Método de Monte Carlo: simulações baseadas em aleatoriedade controlada, úteis para tomada de decisão complexa ou prever melhores ações.

O que estudar

  • Lógica de estados: projetar transições simples e claras.
  • Árvores de comportamento: decompor problemas grandes em blocos reutilizáveis.
  • Algoritmos de busca: principalmente A* e variações para navegação.
  • Steering Behaviors: técnicas para movimento realista (perseguir, fugir, alinhar, separar).
  • PCG: algoritmos de ruído (Perlin, Simplex), L-systems, gramáticas e randomização com regras.
  • Monte Carlo: aprender simulações iterativas para otimizar escolhas da IA em cenários incertos.

Importância

  • Desafio: inimigos adaptativos tornam o jogo mais envolvente.
  • Imersão: NPCs convincentes fazem o mundo parecer vivo.
  • Rejogabilidade: PCG e Monte Carlo criam experiências únicas a cada partida.
  • Escalabilidade: IA modular possibilita desde inimigos simples até sistemas complexos de mundo vivo.

Padrões de Projeto

  • Padrões de Projeto (Design Patterns) são soluções reutilizáveis para problemas comuns de programação.
  • Em jogos, eles ajudam a organizar o código, reduzir dependências e facilitar manutenção e escalabilidade.
  • Não são código pronto, mas modelos de como estruturar partes da aplicação de forma clara e eficiente.

Por que usar Padrões de Projeto?

  • Organização: evita duplicação e bagunça no código.
  • Reutilização: soluções testadas podem ser aplicadas em vários contextos.
  • Comunicação: facilitam o entendimento entre desenvolvedores (nomes conhecidos como Singleton, Observer etc.).
  • Escalabilidade: facilita a evolução de projetos maiores sem quebrar sistemas antigos.

Padrões mais usados em Game Programming

  • Singleton: garante que exista apenas uma instância de um objeto (ex.: Game Manager, Audio Manager).
  • Observer (Event System): permite que objetos se inscrevam em eventos e sejam notificados sem dependências diretas.
  • Factory: cria objetos de forma centralizada, útil para spawn de inimigos, itens e projéteis.
  • Component: organiza comportamentos em módulos independentes que podem ser adicionados/removidos (base da Unity e UE).
  • State: gerencia estados de forma clara (ex.: menu, jogando, pausado, morto).
  • Strategy: troca algoritmos ou comportamentos em tempo de execução (ex.: diferentes padrões de ataque de inimigos).
  • Command: encapsula ações em objetos, útil para inputs, sistemas de undo/redo e gravação de ações.

Importância em jogos

  • Facilitam testes e debugging ao isolar responsabilidades.
  • Reduzem acoplamento, deixando sistemas mais independentes.
  • Tornam o jogo expansível sem precisar refatorar toda a base de código.

Otimização e Performance

  • Otimização e Performance são etapas fundamentais no desenvolvimento de jogos para garantir que o jogo rode de forma fluida, estável e com boa experiência para o jogador.
  • O foco é reduzir uso de CPU, GPU e memória sem comprometer a jogabilidade ou a qualidade visual.
  • Nem sempre otimizar significa deixar “mais rápido”, mas sim equilibrar custo e benefício entre desempenho e qualidade.

Boas práticas

  • Profiling primeiro: medir onde estão os gargalos antes de otimizar.
  • Evitar loops desnecessários: reduzir cálculos repetidos em cada frame.
  • Pooling de objetos: reutilizar instâncias (inimigos, projéteis) em vez de criar/destruir constantemente.
  • Culling e LOD: renderizar apenas o que está visível (occlusion culling) e usar níveis de detalhe em modelos distantes.
  • Batching de draw calls: agrupar objetos semelhantes para reduzir chamadas de renderização.
  • Uso eficiente de física: simplificar colisores, limitar cálculos complexos, usar triggers quando possível.
  • Gerenciamento de memória: descarregar assets não usados, comprimir texturas, evitar vazamentos de memória.
  • Multithreading: distribuir tarefas pesadas (IA, carregamento de dados) em threads separadas.

Áreas comuns de otimização

  • Gráficos: shaders otimizados, menos polígonos, iluminação bem configurada.
  • Scripts: reduzir chamadas no Update/ Tick, preferir eventos e timers.
  • Áudio: comprimir arquivos, usar streaming em vez de carregar tudo na memória.
  • Rede: minimizar tráfego de dados, compressão de pacotes, sincronização eficiente.

Importância

  • Experiência do jogador: frame rate estável e tempo de resposta rápido.
  • Compatibilidade: jogos que rodam bem em dispositivos mais fracos alcançam mais público.
  • Escalabilidade: otimizações permitem adicionar novos recursos sem quebrar performance.

Controle de Versão

  • Controle de versão é o sistema que registra mudanças no código/arquivos ao longo do tempo, permitindo voltar versões, trabalhar em equipe e rastrear histórico.
  • Em jogos, é crucial para lidar com muitos arquivos (código + assets), colaborar com segurança e evitar perda de trabalho.
  • Particularmente, SVN é mais sólido para o desenvolvimento de jogos, já que ele lida melhor com arquivos binários, e jogos tem muitos arquivos binários, mas ambas as soluções para controle de versão são válidas.

Git (distribuído)

  • Modelo: cada dev tem um repositório completo local; commits offline e push/pull para remotos.
  • Fluxos comuns: Git Flow, Trunk-Based, feature branches, code review via PR/MR.
  • Comandos básicos: git init, git clone, git add, git commit, git push, git pull, git branch, git checkout/switch, git merge, git rebase.
  • Binários grandes: usar Git LFS para uassets, texturas, áudio, etc.
  • Quando usar: equipes distribuídas, necessidade de branches/PRs rápidos e integrações com GitHub/GitLab/Azure.

SVN (centralizado)

  • Modelo: um servidor central com trunk/branches/tags; commits dependem do servidor.
  • Fluxo comum: trabalhar no trunk e criar branches para features/lançamentos.
  • Comandos básicos: svn checkout, svn update, svn add, svn commit, svn status, svn revert, svn merge, svn branch/tag (via svn copy).
  • Quando usar: times que preferem controle central e histórico linear; pipelines legados.

Boas práticas

  • .gitignore / svn:ignore: ignorar pastas geradas (Unreal/Unity): Binaries, Intermediate, DerivedDataCache, Saved, Library, Temp, Logs.
  • Atomic commits: mudanças coesas + mensagens claras (feat/fix/chore + contexto).
  • Branches por feature: isolar trabalho; PR com revisão e testes.
  • Merge frequente: reduzir conflitos (especialmente assets binários).
  • Locks quando necessário: para arquivos binários não mescláveis (modelos, levels) — SVN tem suporte nativo; em Git, combine LFS + lock.
  • Tags/Releases: marque builds estáveis (RC, v1.0, hotfix).
  • Automação: CI para builds/testes; hooks para checagens (linters, tamanho de arquivo, naming).
